## 第 1 階段: 檢索品質 (正確性)

### 1.1 向量檢索準確度
- [ ] **Recall@K 指標** (微軟研究院的 Robustness-δ@K)
  - 測試: 建立 100 筆帶有已知正解的合成查詢 (synthetic queries)
  - 測量: 計算 Recall@5、Recall@10、Recall@20
  - 驗收標準:
    - Recall@10 ≥ 0.85 (找回 85% 之預期結果)
    - Robustness-0.5@10 ≥ 0.90 (90% 之查詢達到 ≥50% 的 recall)
  - 原因: 平均 recall 往往掩蓋了尾端表現；穩健性 (robustness) 才能捕捉困難查詢的問題
  - 參考: Wang et al. (2025) "Towards Robustness" - arxiv.org/html/2507.00379v1

- [ ] **餘弦相似度準確性 (Cosine Similarity)**
  - 測試: 比較計算出的相似度與手動計算之差異
  - 測量: 50 對向量，驗證 ±0.001 精準度
  - 驗收標準: 容許誤差內 100% 吻合
  - 參考: `utils.ts` (cosineSimilarity function)

- [ ] **BM25 關鍵字評分**
  - 測試: 查詢 "Nginx 502" 時應將 "Nginx 502 proxy_buffer_size" 排名高於 "502 error"
  - 測量: 精確符合 (exact-match) 與部分符合 (partial-match) 的排名位置
  - 驗收標準: 精確符合結果需落在前 3 名
  - 參考: `store.ts:93` (bm25LikeScore)

### 1.2 混合檢索平衡
- [ ] **Vector 與 BM25 權重微調**
  - 測試: 於相同的查詢下設定 vectorWeight=[0.5, 0.7, 0.9]，bm25Weight 則相反分配
  - 測量: 前 5 名結果之重疊性、相關性排名
  - 驗收標準: 
    - vectorWeight=0.7 能夠產出平衡結果 (語意 + 關鍵字)
    - 不能有任一極端權重不預期地壓制結果
  - 原因: 混合檢索需要經驗型調諧，預設的 0.7/0.3 未必適應所有領域

- [ ] **最小評分門檻 (Min Score Threshold) 之效果**
  - 測試: 使用 minScore=[0.05, 0.2, 0.5] 進行查詢
  - 測量: 結果數量、排名最差結果的品質
  - 驗收標準:
    - minScore=0.2 能夠過濾雜訊而不遺失相關數據
    - 結果總數隨著 minScore 之提高需成反比 (單調遞減)

### 1.3 語意相關性 (領域特化)
- [ ] **基於分類的過濾 (Category-Based Filtering)**
  - 測試: 儲存帶有 [preference, fact, decision, entity, other] 類別之紀錄
  - 測量: 查詢 "Nginx config" → 驗證 decision/fact 紀錄排名高於 preferences
  - 驗收標準: 相關分類必須落在前 3 名結果內
  - 原因: 對於領域特化記憶，分類(Category)應該要能影響排名

- [ ] **重要性加權 (Importance Weighting)**
  - 測試: 在同一內文但 importance=[0.1, 0.5, 0.9] 之下
  - 測量: 高重要性與低重要性紀錄之排名位置
  - 驗收標準: 高重要性紀錄更常出現在前 5 名中
  - 注意: 雖然目前的實作尚未應用重要性至檢索評分，未來會加進來

  - 測量: 去確認一下最長遠過氣的 50 筆將之捨棄
  - 驗收標準: 碩果尚存留者全是離現在時間最親近新創的內容

### 3.5 注入防護力把關機制 (Injection Control)
- [ ] **固定模式 (預設行為)**
  - 測試: 設定定成 `injection.mode=fixed` 後查回總量高達 10 的記憶
  - 測量: 倒回灌到系統情境裡的實質發行數目狀況
  - 驗收標準: 必須不多於 ≤ `maxMemories` (原廠一般設 3) 這個上限配額設定的量被發放回去。
  - 參考: `injection.ts` (fixed mode logic)

- [ ] **預算模式 (Budget Mode)**
  - 測試: 將其設定在 `injection.mode=budget` 以及額度最高只發 `budgetTokens=1000`
  - 測量: 所有被丟還出來注射的這些記憶其整體之「估重換算的 token 體積」合計情況。
  - 驗收標準: 回饋不違逆總體算術 ≤ `budgetTokens` 之數。
  - 參考: `injection.ts` (budget mode logic)

- [ ] **自適應排外模式 (Adaptive Mode)**
  - 測試: 排列定檔之時 `injection.mode=adaptive` 又搭配參數 `scoreDropTolerance=0.15`
  - 測量: 前後兩位各連續不中斷而接著緊追之下一筆回饋進場時之積分大跌 (Score drop) 分區差異
  - 驗收標準: 當前筆跌率大於跳水深度 > 15% 時將收手暫停放人進關之操作動作。
  - 參考: `injection.ts` (adaptive mode logic)

- [ ] **縮寫摘要能力調適區**
  - 測試: 改編成為 `injection.summarization=auto` 型態後並且強去調一個偏長型的長篇文回灌。
  - 測量: 終產對應 `summaryTargetChars` 看文數字到底多長。
  - 驗收標準: 已摘要去蕪存菁下的該新濃縮文件必然為 ≤ `summaryTargetChars` 指定數之體積。
  - 參考: `summarizer.ts` (summarization logic)

- [ ] **Code 程式碼摘要作業**
  - 測試: 利用改制 `injection.codeSummarization.mode=preserve` 下執行讀回某一段屬於存留程式部分之回憶。
  - 測量: 截短斬短後的 Code 那括號有沒有平穩之保持成對封閉平衡對數之穩定架勢。
  - 驗收標準: 於縮短過之語法後還是有正確合規的語法邏輯形式面貌外形。
  - 參考: `summarizer.ts` (code truncation logic)

## 第 4 階段: 效能與擴展性

### 4.1 延遲指標評測 (Latency Benchmarks)
- [ ] **搜尋回傳遲延之延時反應 (於 p50 與 p99 分布)**
  - 測試: 單在具 10K 筆大存貨資料之庫底下抽查做約 1000 遍探問找驗作業操作。
  - 測量: 量出 p50 與 p99 居間數的運營秒數差指標數值。
  - 驗收標準:
    - p50 位於 < 100ms (走 hybrid 檢索架構)
    - p99 不高過 < 500ms
  - 原因: 工具需反應迅速；尾部延遲對於 UX 體驗十分重要
 
- [ ] **置入塞進時間延時**
  - 測試: 一百次的接連序貫注入插入執行工作
  - 測量: 單次一般所需消耗常態時數值平均
  - 驗收標準: 平均每項動作在 50ms 之下完成

- [ ] **擷取整體 Scope 時間延時**
  - 測試: 把某定 Scope 下之 1000 項物件列隊調集抓取而下
  - 測量: 拿出來之並將之調排列好之全程流逝走動鐘錶長值數
  - 驗收標準: 落於 200ms 以內為佳

### 4.2 儲存耗費比

- [ ] **Vector 容量空間與效力存放比例**
  - 測試: 利用內含有維度均 384 的 10K 量資料下去壓存放做。
  - 測量: 實際耗去佔有的硬碟實體消耗位元。
  - 驗收標準: 佔用空間應在小於 50MB (< 50MB) 尚在對開發平台合理的幅度之下。
  - 注意: 理順的利用壓縮去管這件事也需檢視有無發生 memory leaks 不慎洩露系統主區 RAM 等不良事蹟產生。

- [ ] **索引架構下本來會生多餘體積之增胖指數比率**
  - 測試: 有外加放著帶了 vector index 跟純空著時，比較出來二者的耗費。
  - 測量: 其變大出額外負載的變胖百分比率差距。
  - 驗收標準: 其累贅不超額逾越在20% (> 20% overhead) 多出這數的紅線上。

### 4.3 重載彈性容受延展極限能耐極點分析
- [ ] **全境無涯界線滿放之總額可供量極限**
  - 測試: 針對一筆獨立的儲存域，強將硬餵進 100K 以上總筆量。
  - 測量: 對此進行尋問操作其搜尋時滯的降效退化指數評量差分。
  - 驗收標準: 此刻的 p50 平均尋問延遲需力保不致過慢小於 500ms 即視為具退化合理不失格等次。
  - 注意: 所謂 Pruning 之強迫割棄修剪在真實世界上運作即可對這問題有效止血之用。

- [ ] **最多同時擁存多少區間的各門 Scope 目錄容受力**
  - 測試: 強硬開啟約出造下共高達 1000 條全各自不同獨立專屬 project 專用域區目錄名下
  - 測量: scope 調用與解析切換尋找定址耗時分量。
  - 驗收標準: 分區轉換鎖住目標之 Lookup 步驟不能高逾單趟 10ms 限制。

### 4.4 效能檢核關卡硬性門戶設防標水切口 (Benchmark Thresholds)

這個層級是在實質宣導延遲門戶驗看的一種標準規範防洪政策表。

#### 硬性閘口防線 (Hard Gates)
若是執行測試命令打下 `npm run benchmark:latency` 而其表現低落甚至踩到了紅標之底處即判定測不過:
- `search.p50 < 100ms`
- `search.p99 < 500ms`

歸論原理剖析之:
- 這二門關鍵值數據就是對接 Phase 4.1 這一部要求點上。
- 為在第一時間裡滿足發布週期中的響應 UX 所作出的必要標尺保固。

#### 資訊化參考之監視指標點
僅於分析列表裡做成標示報告，未在指令防堵判定線被利用阻擋關門:
- `insert.avg < 50ms`
- `list.avg < 200ms`

歸論原理解說:
- 因為此二項寫出跟讀遍操作只為一般平時作業作維修查估觀察指標警示，唯現行當下只重心全數防守搜索(Search)速度才當釋出版放水標竿門的鐵網。
- 不過爾後若是指標愈來愈精準趨於更可預估準確的地步，這些將也被升等到歸入成硬門檻防守行列。

#### Benchmark 分析組設定型態配置方案 (Profile)
- 原生預設出貨用 `release` 基本標準配版組設定:
  - 會做搜尋指標測試: 用有 1000 筆資料者發動連 200 項測試跑分
  - 寫進指標: 連跑一百動不停加寫之接力
  - 定項區目錄名 Scope 調閱抓出測試: 對大受限 `1000` 者去弄 10 趟大操作
- 如果使用可額外選進切到之全套 `full` 硬實力考場配置版的話:
  - 在針對破高達 10000 筆之巨型區進行高標配的一千題之查找
  - 在寫入仍舊 100
  - 對目錄查搜則依然是在設定上限一千點的做 10 次跑動

- 這個出廠設置之預設檔名為 release 本來就是要又快又順的一回過之保證門方案，利於平時本機快速操作測試及放置進小規模容器檢定來用就好。

### 來計計幫這來量打算算出 Recall@K 這計量之給得算出分出點數去幫之機計算算打力計數算機 (Recall@K Calculator)
```typescript
function calculateRecall(retrieved: string[], groundTruth: string[], k: number) {
  const topK = retrieved.slice(0, k);
  const matches = topK.filter(id => groundTruth.includes(id)).length;
  return matches / k;
}

function calculateRobustness(recalls: number[], delta: number) {
  const passing = recalls.filter(r => r >= delta).length;
  return passing / recalls.length;
}
```

### 給這測度探在看延誤大走跑算多會久慢之之測探久去久在慢拖器把分計測等久探計時計測秒去指度之(Latency Profiler)
```typescript
async function profileLatency(fn: () => Promise<any>, iterations: number) {
  const times = [];
  for (let i = 0; i < iterations; i++) {
    const start = performance.now();
    await fn();
    times.push(performance.now() - start);
  }
  times.sort((a, b) => a - b);
  return {
    p50: times[Math.floor(times.length * 0.5)],
    p99: times[Math.floor(times.length * 0.99)],
    avg: times.reduce((a, b) => a + b) / times.length,
  };
}
```

- **看那個等尾大掉其尾不掉與否那的延遲防線標看尾巴才真 (Monitor Tail Performance)**: 要看這個延遲是那些有掉速到其大有 p99 處的那個才最有在觀之的。比其拉低就這總把的平均更為重要啊對這有給交互往來時的。
- **那有防絕防跨之把各區範圍做限做死這才屬最要命處處之最重 (Scope Isolation is Critical)**: 由於若做不到多數專案要各自支援上不被跨不漏出那之等保固就是要把各防隔做實才可為才。
- **與等那等鑲等提供搭之包那要把掛各提供大主都作虛架與掛等掛能為防在先這都 (Embedding Provider Abstraction)**: 在造此測下就一當同把造掛這造它這可與其這多後能接上這之不同這其等在各提供大咖等可同在此測掛此用(在此後會有 OpenAI啦，或等等它與及在地跑等的多這將都納有為其可)。
